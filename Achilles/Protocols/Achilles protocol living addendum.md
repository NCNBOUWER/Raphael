# Achilles protocol living addendum

Purpose: clarify that `Achilles protocol.md` is a current control surface, not a hard final protocol.

## Status

The Achilles protocol is living. It should expand as older chats, open threads, workbook versions, Drive folders, source documents, agent files, and archive records are consolidated.

## Current correction

The protocol should not be read as final or rigid. It is the current operating baseline. When better or more complete standards are found in prior material, they should be integrated into the protocol, package maps, task registers, and canonical files.

## Broader scope

Achilles is not Raphael-specific. Raphael is the current priority package, but Achilles is the cross-project consolidation layer for:

- current files;
- older files;
- open chats and strings;
- thought trains;
- workbook versions;
- Drive folders;
- Git repositories;
- agent task files;
- archive registers;
- final print and publish surfaces.

The secondary Z stack remains deferred for now. Do not set Z-stack automations yet.

## Archive interpretation

Do not treat archive work as a storage move only. The correct sequence is:

1. inspect the older item;
2. extract useful context, decisions, tasks, and source material;
3. update the correct canonical or current save file;
4. add unresolved material to open tasks;
5. place the old item in the archive register;
6. only later place fully extracted items into the approval-gated disposal register.

## Current response discipline

Every Achilles continuation should still report:

```text
Completed:
Blockers:
Next steps:
Owner decisions required:
Files/registers updated:
```

## Expansion requirement

Whenever a recurring standard, workflow, package boundary, or term is found in older source material, add it to the protocol or a protocol addendum instead of relying on memory.
