# Fascicle III build questions

Purpose: lock build expectations before Raphael integration work begins.

## Critical questions

1. **Raphael source set:** Which exact files are authoritative for Fascicle III: private PDFs, Drive workbooks, GitHub Raphael source, all of these, or a named subset?

2. **Equation exposure:** May Fascicle III display exact Raphael equations in Git, or should Git use equation IDs and descriptions only while exact forms stay in Drive/private source?

3. **Raphael Suite scope:** Should the future `Raphael Suite` be a Markdown call registry only for now, or should it include a minimal Python interface skeleton after Fascicle III is complete?

4. **Claim strength:** Should Fascicle III use conservative labels (`support`, `constraint`, `falsification`) or allow stronger internal labels (`proof`, `derivation`, `validation`) when inherited from Fascicle I-II?

5. **Fidelity default:** Should Raphael-facing claims begin at `110%` only when they are fully inherited from Fascicle I-II, while all source-governed or physical claims remain below empirical thresholds until tested?

6. **Public/private split:** Should final public-facing GeoMatrices keep only clean summaries and move the full crosswalk/falsification registers into `INTERNAL/`?

7. **Workbook sync:** Which workbook should receive the Fascicle III task state first: Internal, Master, Public, or only the live `OPEN_TASKS` register until all three fascicles are merged?

8. **EML#F usage:** Should Fascicle III mention EML#F only in an external-source appendix, or should EML#F be linked as a possible derivation-library interface for future Raphael Suite calls?

9. **ToE terminology:** Should Fascicle III avoid `ToE proof` wording until final GeoMatrices merge, or allow internal `ToE candidate framework` phrasing?

10. **Final merge rule:** After Fascicle III, should Fascicles I-II-III remain as build logs under `N^3/`, with the final live system rebuilt as `GeoMatrices/`, or should `N^3/` itself become the final GeoMatrices folder?

## Inferred defaults if not overridden

1. Use all available Raphael source material, but keep source hierarchy explicit.
2. Use equation IDs/descriptions in Git unless exact equations were already approved for publication.
3. Keep Raphael Suite as Markdown call registry only until after Fascicle III.
4. Use conservative public labels, stronger internal labels only where proof class justifies them.
5. Use `110%` only for inherited mathematical/geometric proof stacks; empirical claims require evidence.
6. Public summaries outside; full crosswalk/falsification inside `INTERNAL/`.
7. Use `OPEN_TASKS` first; workbook sync later and in place.
8. Mention EML#F as external-source appendix/interface candidate only.
9. Use `ToE candidate framework` until final merge and owner sign-off.
10. Keep fascicles as build logs and create a final GeoMatrices live surface later.
